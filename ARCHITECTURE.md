# Архитектура LLM-Wiki (Персональный Экзокортекс)

Эта система — гибрид идей Карпатого (простота), Brad Groux (происхождение данных) и Farzaa (смысловая структура). 
Цель: создать «карту разума», которая накапливает знания, связи, паттерны и решения, а не просто складирует файлы.

## 1. Базовые принципы

* **Тупые файлы, Умная модель:** Никаких векторных баз данных (SQLite), RAG-пайплайнов и сложных обвязок. Современные LLM (Gemini 1.5/3.1 Pro, Claude 3.5/4.6 Sonnet) вмещают миллионы токенов. Мы храним всё в простом Markdown, а автономный агент читает и правит файлы напрямую.
* **Lineage (Происхождение):** Жесткая связность. Любое утверждение, концепт или факт в папке `wiki/` должно иметь ссылку (`[[wikilink]]` или footnote) на оригинальный сырой файл в `raw/`. Если через год факт вызывает сомнения — всегда есть путь к исходнику.
* **Писатель, а не Клерк:** Агент не просто дописывает новые факты в конец файла (как лог). При добавлении новой информации агент *переписывает* статью так, чтобы она читалась как связный текст (по темам, а не по хронологии).
* **Стиль Википедии про одного человека:** Статьи пишутся нейтральным, сухим языком. Никаких оценочных суждений агента. Эмоциональный вес передается исключительно через *прямые цитаты* из сырых заметок.
* **Смысл важнее энциклопедичности:** Статья про книгу — это не пересказ сюжета, а ответ на вопрос «как эта книга повлияла на пользователя, его проекты или мышление».

## 2. Структура Репозитория

```text
llm-wiki/
├── raw/                ← Только для чтения. Сырые заметки, фото, дампы сообщений, дневники.
├── wiki/               ← Скомпилированные знания. Агент (ИИ) полностью владеет этой папкой.
│   ├── projects/       ← Проекты (от задумки до результата)
│   ├── philosophies/   ← Принципы работы и жизни
│   ├── patterns/       ← Паттерны поведения (триггер -> реакция -> результат)
│   ├── decisions/      ← Важные решения (контекст, варианты, почему выбрано)
│   ├── transitions/    ← Переходные/лиминальные периоды
│   ├── eras/           ← Эпохи жизни
│   ├── people/         ← Люди (роль и влияние)
│   ├── tensions/       ← Конфликты ценностей или приоритетов (trade-offs)
│   └── concepts/       ← Остальные смысловые узлы
├── AGENTS.md           ← Главная инструкция для ИИ (правила форматирования, теги, логика)
├── index.md            ← Мастер-индекс всех статей (собирается агентом)
└── log.md              ← Append-only лог действий агента
```

## 3. Топология Синхронизации

Интеграция ИИ-агента в личную базу знаний требует решения проблемы синхронизации между мобильным устройством, десктопом и сервером/облаком, где работает ИИ.

### Вариант А: Pure Git (Бесплатный)
Все устройства работают с единым Git-репозиторием.
* **Схема:** Mobile (Working Copy + Obsidian) <-> GitHub <-> Desktop (Obsidian Git) <-> AI Agent.
* **Особенности:** Требует настройки Git-клиентов на мобильных устройствах. Возможны конфликты слияния, если ИИ-агент пушит изменения в тот момент, когда пользователь редактирует файл на телефоне без интернета.

### Вариант Б: Гибридный (Obsidian Sync + Git Bridge)
Разделение контуров: человеческий (нативный Sync) и машинный (Git).
* **Human Layer (Obsidian Sync):** Телефон и Десктоп общаются мгновенно через нативный платный Obsidian Sync. Никаких мерж-конфликтов. Пользователь просто кидает мысль в `raw/` на ходу.
* **AI Layer (Git Bridge):** Десктоп с помощью плагина Obsidian Git выступает как мост — делает авто-пуш папок в приватный Git-репозиторий. ИИ-агент (сервер) забирает файлы оттуда, работает как ИИ-писатель, коммитит переписанные статьи в `wiki/` и пушит обратно в GitHub. Десктоп подтягивает изменения из Git, и нативный Sync тут же закидывает готовую статью обратно на телефон.

## 4. Жизненный цикл знаний (Workflow)

1. **Ingest (Сбор):** Добавление сырых данных (текст, аудио, фото) в папку `raw/`. Это единственное ручное действие пользователя.
2. **Absorb (Поглощение):** ИИ-агент (по cron-задаче или триггеру) читает новые файлы в `raw/`, анализирует `index.md`, понимает смысл новой информации и обновляет/создает статьи в `wiki/`, проставляя ссылки (Lineage). Завершается `git push`.
3. **Query (Запрос):** Для ответов на сложные вопросы агент загружает всю папку `wiki/` в контекст модели (без использования RAG) и синтезирует ответ с цитатами.
4. **Cleanup (Рефакторинг):** Периодический проход агента по базе для дробления слишком больших статей (>150 строк), поиска "сирот" и выделения повторяющихся тем в новые концепты.
