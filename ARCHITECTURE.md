# LLM-Wiki Architecture (Personal Exocortex)

This system is a hybrid of ideas from Karpathy (simplicity), Brad Groux (data lineage), and Farzaa (semantic structure).
Goal: build a "mind map" that accumulates knowledge, connections, patterns, and decisions — not just stores files.

## 1. Core Principles

* **Dumb Files, Smart Model:** No vector databases, RAG pipelines, or complex tooling. Modern LLMs (Gemini 1.5/3.1 Pro, Claude 3.5/4.6 Sonnet) can handle millions of tokens. We store everything in plain Markdown, and an autonomous agent reads and edits files directly.
* **Lineage:** Strict traceability. Every statement, concept, or fact in `wiki/` must have a link (`[[wikilink]]` or footnote) to the original raw file in `raw/`. If a fact is ever questioned, there is always a path back to the source.
* **Writer, Not a Clerk:** The agent does not simply append new facts to the end of a file (like a log). When new information is added, the agent *rewrites* the article so it reads as a coherent text (organized by theme, not chronology).
* **Wikipedia Style for One Person:** Articles are written in a neutral, dry tone. No editorial judgments from the agent. Emotional weight is conveyed exclusively through *direct quotes* from the raw notes.
* **Meaning Over Encyclopedia:** An article about a book is not a plot summary — it answers the question "how did this book influence the user, their projects, or their thinking?"

## 2. Repository Structure

```text
llm-wiki/
├── raw/                <- Read-only. Raw notes, photos, message dumps, diaries.
│   └── entries/        <- One .md per entry (generated by ingest)
├── wiki/               <- Compiled knowledge. The AI agent owns this folder entirely.
│   ├── projects/       <- Projects (from idea to outcome)
│   ├── philosophies/   <- Work and life principles
│   ├── patterns/       <- Behavioral patterns (trigger -> reaction -> outcome)
│   ├── decisions/      <- Important decisions (context, options, reasoning)
│   ├── transitions/    <- Transitional/liminal periods
│   ├── eras/           <- Life epochs
│   ├── people/         <- People (role and influence)
│   ├── tensions/       <- Value conflicts or priority trade-offs
│   ├── concepts/       <- Other semantic nodes
│   ├── _index.md       <- Master index with aliases (built by the agent)
│   └── _backlinks.json <- Reverse link index
├── AGENTS.md           <- Main AI instruction file (formatting rules, tags, logic)
└── ARCHITECTURE.md     <- System architecture and design rationale
```

## 3. Sync Topology

Integrating an AI agent into a personal knowledge base requires solving the sync problem between a mobile device, desktop, and the server/cloud where the AI runs.

### Option A: Pure Git (Free)
All devices work with a single Git repository.
* **Diagram:** Mobile (Working Copy + Obsidian) <-> GitHub <-> Desktop (Obsidian Git) <-> AI Agent.
* **Notes:** Requires setting up Git clients on mobile devices. Merge conflicts are possible if the AI agent pushes changes while the user is editing a file on their phone offline.

### Option B: Hybrid (Obsidian Sync + Git Bridge)
Separation of layers: human (native Sync) and machine (Git).
* **Human Layer (Obsidian Sync):** Phone and Desktop communicate instantly via native paid Obsidian Sync. No merge conflicts. The user simply drops a thought into `raw/` on the go.
* **AI Layer (Git Bridge):** The Desktop, using the Obsidian Git plugin, acts as a bridge — auto-pushing folders to a private Git repository. The AI agent (server) pulls files from there, works as an AI writer, commits rewritten articles to `wiki/`, and pushes back to GitHub. The Desktop pulls changes from Git, and native Sync immediately sends the finished article back to the phone.

## 4. Knowledge Lifecycle (Workflow)

1. **Ingest (Capture):** Adding raw data (text, audio, photos) to the `raw/` folder. This is the only manual action required from the user.
2. **Absorb (Compilation):** The AI agent (via cron job or trigger) reads new entries in `raw/entries/`, analyzes `wiki/_index.md`, understands the meaning of the new information, and updates/creates articles in `wiki/`, adding source links (Lineage). Finishes with `git push`.
3. **Query:** To answer complex questions, the agent loads the entire `wiki/` folder into the model's context (without using RAG) and synthesizes an answer with citations.
4. **Cleanup (Refactoring):** Periodic agent pass through the base to split oversized articles (>150 lines), find orphans, and extract recurring themes into new concepts.
